## Finding 1 — Remote command injection in cloudflare-nginx-setup.sh (HIGH)

**Vector.** Pre-fix script accepted `<DOMAIN> [SERVER_IP] [WEBROOT_NAME]` and
interpolated all three values into the remote command body of an `ssh ... root@$SERVER "...${DOMAIN}...${WEBROOT}..."` invocation without input
validation. A malicious DOMAIN like `'foo.com$(touch /tmp/pwned)'` would execute
arbitrary commands as root on the WWW server during nginx provisioning.

**Fix (`9b1d9c0`):**
- Strict regex validation gate before any outbound call:
  - `DOMAIN_RE='^[a-z0-9]([a-z0-9-]*[a-z0-9])?(\.[a-z0-9]([a-z0-9-]*[a-z0-9])?)+$'`
  - `IP_RE='^[0-9]+(\.[0-9]+){3}$'`
  - `NAME_RE='^[a-zA-Z0-9_.-]+$'` + explicit `..` rejection
- Remote nginx vhost creation uses stdin heredoc + remote-side re-validation
  (defence in depth). DOMAIN and WEBROOT_NAME are passed via env to the
  remote shell, never interpolated into the SSH command string.
- Switched to `deploy@` user (narrow `sudo`) and dropped redundant `ssh` flags.

**Regression test.** `finding-1-cloudflare-nginx-injection.bats` mocks
`ssh / scp / curl / openssl` and asserts that for malicious input the script
exits non-zero **before** any mock is invoked (`calls.log` is empty).

## Finding 2 — Python heredoc injection in cloudflare-nginx-setup.sh (HIGH)

**Vector.** Pre-fix script used `python3 - <<PY` (unquoted terminator). The
shell interpolated `${DOMAIN}` and `${CF_TOKEN}` directly into the Python
source before execution; a domain string containing Python syntax would have
executed arbitrary Python with elevated trust.

**Fix (delivered together with Finding 1 in `9b1d9c0`; regression test added
in `7ed349b`).**
- Quoted heredoc terminator `<<'PY'` blocks shell expansion inside the body.
- Dynamic values flow via `os.environ['DOMAIN']` / `os.environ['CF_TOKEN']` /
  `os.environ['CF_API']` / `os.environ['TMP']`.

**Regression test.** Two static-grep assertions: every `<<PY` heredoc-open
line is single-quoted, and the heredoc body uses `os.environ[...]` and
contains no `${DOMAIN}`/`${CF_TOKEN}`/`${CF_API}`/`${TMP}` interpolation.

## Finding 3 — SSH host-key verification bypass (MEDIUM)

**Vector.** `skills/infra-automation.md` shipped `ssh -o
StrictHostKeyChecking=no -o BatchMode=yes ...` as the canonical batch
recipe. Because skills are executable instructions for AI agents, the
unsafe pattern was being copied into every consumer playbook.

**Fix (`e7248d8`).**
- Documented one-time bootstrap via `ssh-keyscan -H "$host" >> ~/.ssh/known_hosts`
  with a pointer to `Areas/Infrastructure/known-hosts-rotation.md`.
- Canonical batch recipe relies on default `StrictHostKeyChecking=ask` plus
  `BatchMode=yes` (fails fast on unknown host).
- Switched to `deploy@` user.
- Wrapped the unsafe original pattern in a
  `<!-- security:counter-example -->...<!-- /security:counter-example -->`
  fence so the pedagogical reference is explicit and skipped by the test.

**Regression test.** `finding-3-ssh-host-key-bypass.bats` strips
counter-example fences and scans `skills/agents/commands/templates` for any
remaining `StrictHostKeyChecking\s*=\s*no` literal.

## Finding 4 — OAuth token + venv on `/tmp/` (MEDIUM)

**Vector.** Pre-fix `ga4-admin.md` placed both the python venv
(`/tmp/ga4-tools`) and the OAuth refresh token (`/tmp/ga4-token.json`,
default mode 0644) on `/tmp/`. Race-prone, world-readable, world-writable.

**Fix.** Token storage was migrated to XDG_STATE_HOME by SEC-0001
(`304cc25`): `~/.local/state/datarim/ga4-token.json`, dir mode 0o700, file
written via `os.open(O_WRONLY|O_CREAT|O_EXCL, 0o600)`. This commit (`f46ba6e`)
finishes Finding 4 by:
- Moving the python venv from `/tmp/ga4-tools` to `$GA4_VENV` (default
  `$HOME/.cache/ga4-tools`).
- Replacing brand-named `ARCANADA_CREDS_DIR` env var with generic
  `PROJECT_CREDS_DIR` per `~/arcanada/CLAUDE.md` § Deployment & Credentials.

**Regression test.** Five assertions: `/tmp/ga4-token` absent, `/tmp/ga4-tools`
absent, `os.open` with `0o600` + `O_EXCL`, `os.makedirs` with `0o700`, and
`PROJECT_CREDS_DIR` (no `ARCANADA_CREDS_DIR`).

## Finding 5 — Hardcoded OAuth Client ID in shipped skill (MEDIUM)

**Vector.** `skills/utilities/ga4-admin.md` lines 3 + 15 (commit `077137c2` /
TUNE-0005, 2026-04-17) contained a literal
`client_secret_336808097146-7ai13flkkt176bg6n48tdsiqj442aboc.apps.googleusercontent.com.json`.
Public repo `Arcanada-one/datarim` therefore exposed an OAuth Client ID.

**Fix.** SEC-0001 sanitized HEAD (`304cc25`), rotated the Google Cloud OAuth
client (Phase 2), scrubbed git history via `git filter-repo --replace-text`
+ force-push, and wired a gitleaks CI gate (`e74ded6`). This commit (`9271b23`)
adds a regression test that scans `skills/agents/commands/templates/docs` for
any literal `client_secret_<digits>-<chars>.apps.googleusercontent.com`
filename outside of `<!-- security:counter-example -->` fences, and that
`ga4-admin.md` keeps the canonical `glob.glob('client_secret_*...')` pattern.

**Regression test.** `finding-5-hardcoded-client-id.bats` (2 checks).

## Finding 6 — `curl | bash` install recipe (MEDIUM)

**Vector.** `skills/ai-quality/deployment-patterns.md:141` shipped
`RUN curl -fsSL https://claude.ai/install.sh | bash` as the canonical
Dockerfile recipe for CLI connector services. No integrity check; an
upstream installer swap would silently propagate to every consumer build.

**Fix (`051fc86`).**
- Replaced the unsafe one-liner with an ARG-driven hash-pinned download:
  `CLAUDE_CLI_VERSION` + `CLAUDE_CLI_SHA256`, `curl -fsSL -o /tmp/x.tgz` →
  `echo "${SHA256}  /tmp/x.tgz" | sha256sum --check` → `tar -xzf` (Renovate /
  Dependabot bumps version + hash in lockstep).
- Added `npm ci && npm audit signatures` alternative for npm-distributed CLIs.
- Wrapped the original pattern in a `security:counter-example` fence as
  pedagogical reference.

**Regression test.** `finding-6-curl-bash-install.bats` (2 checks): no
unfenced `curl|bash`/`curl|sh`/`wget|bash`/`wget|sh` in shipped artifacts; the
hash-pinned `sha256sum --check` alternative is documented.
